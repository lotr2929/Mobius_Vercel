### Diagnosis of 2D Compass Widget Alignment and Rotation Issues

The core of the problem lies in the reconciliation of two distinct north definitions (`designNorthAngle` and `globalNorthAngle`) with the visual representations of the compass body and its True North needle. The reported issues with "alignment and rotation behaviour" suggest a miscalculation or incorrect application of these angles to the respective CSS/SVG transformations.

Based on the provided key variables, here's a systematic breakdown of potential issues:

1.  **Compass Widget Body Alignment (`npRotEl.style.transform`):**
    *   **Purpose:** This CSS transform rotates the entire compass widget element (`npRotEl`) to align it with the *design grid's north*.
    *   **Hypothesis:** The calculation for this transform likely misinterprets `designNorthAngle`. For instance, if `designNorthAngle` represents the angle from True North to Design North, the widget's rotation needs to orient its own "north" marker to point in the direction of `designNorthAngle` relative to the screen's vertical axis. An incorrect sign, offset, or misunderstanding of the CSS rotation origin/direction (e.g., clockwise vs. anticlockwise, or reference axis) would cause misalignment.

2.  **True North Needle Rotation (`tnNeedleEl.setAttribute('transform')`):**
    *   **Purpose:** This SVG transform rotates the True North needle *relative to the compass widget body*.
    *   **Hypothesis:** This is the most complex rotation as it must account for:
        *   The `globalNorthAngle` (True North's offset from the world coordinate system).
        *   The `designNorthAngle` (the design grid's orientation relative to True North).
        *   The current rotation of the compass widget body itself (controlled by `npRotEl.style.transform`).
    *   If the needle points incorrectly, it suggests the calculation does not properly combine these angles. Specifically, the needle's rotation (relative to its parent, the compass body) should effectively *cancel out* the compass body's rotation such that it points towards True North in the global context, possibly factoring in the `globalNorthAngle` which defines True North in the first place. A common error here is forgetting to account for the compass body's own rotation when positioning the needle, or misinterpreting the `globalNorthAngle`'s reference point.

3.  **Design North Overlay (`dnGroupEl` Visibility and Alignment):**
    *   **Purpose:** This arrow indicates the Design North direction, appearing only when `designNorthAngle` differs from `globalNorthAngle`.
    *   **Hypothesis:** If this element shows incorrect visibility or alignment, it points to issues with:
        *   The conditional logic for `designNorthAngle !== globalNorthAngle`.
        *   Its own rotation logic, which should directly reflect `designNorthAngle` (or the relative angle between True North and Design North) to point correctly.
